CPL3 version 2.0.0 released on October 16th, 2017 New Release Roadmap posted October 16th, 2017 by Michael Witt

We are very happy to announce that the CPL3 version 2.0.0 was released today! This major release includes the creation of a new IHS Markit theme and the scoping changes mentioned in our previous blog posts.

New IHS Markit theme

We've been working hard on creating a new IHS Markit branded theme for our CPL3 customers to start using. Our new IHS Markit theme which we have named the "IHSM theme" will become the main theme of the CPL3 from today moving forward.

What themes are now available?
  1. IHSM theme - "styles-nw-ihsm.css"
  2. Default theme - "styles-nw.css"
  3. IHS theme - "styles-nw-ihs.css"
What's happening to the old IHS theme?

Our intent is to leave the IHS theme alone and sunset its use over a transition period of the next 2 to 3 months (or 4 to 6 minor releases of the CPL3 npm package in other words). A specific file in our package called "styles-nw-ihs.css" has now been created that contains the IHS theme. The styles in our default stylesheet named "styles-nw.css" will continue to be the IHS theme styles for a short transition period after which we will change them to start using the new IHSM theme. Please use this transition time to prepare your project to move fully onto the IHSM theme.

How do we use the new IHSM theme?

The new IHSM theme is directly available in the newly created "styles-nw-ihsm.css" file. After the transition period we will override the default styles in the "styles-nw.css" file with the IHSM theme styles from the "styles-nw-ihsm.css" file. Choose the option that best works for you.

  1. Use the "styles-nw-ihsm.css" directly because you're ready for IHSM styles.
  2. Continue to use the "styles-nw.css" to get the IHS styles today and automatically change over to the IHSM styles when they become the default.
  3. Stick with the IHS styles by specifically linking to the "styles-nw-ihs.css" file. You'll change to the IHSM styles when you're ready. (But make sure it is before we remove them from our package!)
Will the IHS theme styles go away?

Yes, they will. At some point we will remove them from our package and cease to maintain them or offer support. We haven't set a hard date for this yet but we expect it will likely be coming with our next major release or a period of 6 months or less - whichever comes first. We will communicate our intentions as soon as it becomes clear what our best course of action is. Please voice any concerns you have on this topic early so we can work with you toward an appropriate resolution.

When the "styles-nw-ihs.css" file is removed from our package there will no longer be a need to distinguish between themes. Thus, we expect to remove the styles-nw-ihsm.css" file as well and simply return to using the "styles-nw.css" file exclusively.

Let's see it in action

Click away … it's fun! Go ahead and give it a try by clicking the theme you want from our new Theme Switcher in the top right corner of the site header. Doing this will also show you what things have changed from one theme to the next.

CPL3 scope changes in version 2.0.0 New Roadmap posted October 10th, 2017 by Michael Witt

Required Breaking change The scoping class cpl-scope is now required in version 2.0.0 going forward.

This change is a strong "line-in-the-sand" kind of change. Once you've updated to version 2.0.0 of the csspatternlibrary3 npm package you will only get our styles within the scoped area of your HTML that has the scoping class cpl-scope. Any HTML outside of this scope will no longer be affected. Any project that updates without making the change will be in a "broken" state.

Who should update?

Projects or applications that meet some of the following criteria are good candidates to immediately update:

  1. Smaller projects where it would be easy to meet our "Post update requirements" noted below
  2. You're simply adding the cpl-scope class to the html or body tags - something like <html class="cpl-scope">
  3. Your project uses all of CPL3 and you can make a small change to your "master page" or outer-most template (usually the "index.html" file) which will quickly get you back into a "working" state.
  4. You've planned and scheduled to be ready to take on this update.

Who should NOT update until they are ready?

Projects or applications that have any of the following issues are probably not ready to update until preparations are made:

  1. Development resources are already committed to other needs within your project and cannot be freed-up to make the necessary changes.
  2. Your project is large or complicated. Thus, making this change needs planning.
  3. You need to pick and choose where to add the cpl-scope class within your project's HTML
  4. Your project only uses pieces of CPL3 and you need to evaluate before proceeding.
  5. You're NOT YET planned, scheduled, or otherwise ready to take on this update. You don't yet know what the impact of it will be on your project. You don't yet know how or where you will be making the changes to add the scoping class cpl-scope to your code.

Post update requirements

After updating to csspatternlibrary3 version 2.0.0 and going forward:

To use our library of styles you must add what we're calling a scoping class to your code. Add a cpl-scope class to the parent element of any sections of your site or application that should receive CPL3 styles. HTML not contained within this class will no longer be effected in any way by the CPL3.

We expect most will simply add this class to the root html element like <html class="cpl-scope">.

If you need to add it elsewhere please make sure that it is added to an html element of its own like <div class="cpl-scope">

DO NOT ADD IT TO HTML ELEMENTS THAT ALREADY HAVE EXISTING CPL-SPECIFIC CLASSES like <div class="row cpl-scope"> because doing so will give unexpected results.

Why was this change made?

See our previous roadmap blog post titled "Major customer complaints, issues, and our plan for action" for more details. Additionally, we know that our package is already on the "bloated" side of things. After evaluating … this solution was chosen because it offers the best path for continuing to trim down the weight of our package while also offering (what we thought was) an acceptable amount of flexibility to those customers that update. This path did not force us to increase the weight of our core stylesheet ("styles-nw.css") by 3 - 6 times which is what our main alternative would have required. That alternative would have provided a more graceful fallback with all the styles mixed in the same stylesheet. Fortunately, our "Transition Options" described below provide most of that graceful fallback with a clear path for removing unused styles when possible.

Transition Options

  1. Delay updating to the 2.0.0 version
  2. Use both stylesheets from the 1.0.12 and 2.0.0 versions
  3. Request special help from the CPL3 team

Using both stylesheets

If you cannot make all the changes necessary to fully update to version 2.0.0 but need to apply it piece-meal to your project consider using both stylesheets. The stylesheet ("styles-nw.css") from version 1.0.12 will allow your project's already working code to continue to work (as it did in version 1.0.12). The stylesheet ("styles-nw.css") from version 2.0.0 will override version 1.0.12 styles whenever a scoping class is appropriately applied.

  • Pro Smooth transition while the update is being integrated.
  • Con Your project will incur the weight of both stylesheets until the 1.0.12 styles can be removed. 
<link rel="stylesheet" href="css/styles-nw-1.0.12.css" media="all"> <!-- TODO: Remove when 1.0.12 styles are no longer needed. -->
<link rel="stylesheet" href="css/styles-nw.css" media="all"> <!-- from 2.0.0 -->

Please start making the appropriate changes to your project to continue to receive updates from the CPL3. Contact Michael Witt with any questions or concerns you have.

CPL3 updates planned for October 13th, 2017 release and beyond New Roadmap posted October 4th, 2017 by Michael Witt

We are currently updating CPL3 daily to get this website up to date. We’re improving the documentation and trying to fix the numerous bugs and broken bits. Major broken items like the “Full Documentation” link in the left rail navigation are definite “black-eyes” for us that we’re trying to fix ASAP, but we do have other higher priorities that are being worked on to meet business needs. This is good news for you because those business needs likely match up with what you want out of the CPL3.

Roadmap

The immediate (over the next 1 - 3 weeks) roadmap includes:

  1. Removing CPL3 leaking styles by scoping the styles of the CPL3. This will limit CPL3 styles to affecting only HTML elements with the cpl-scope class applied. The children elements of this HTML element will of course also be styled by CPL3. See the notes below in the customer complaints section for more information.
  2. Updating the CPL3 to have both IHS and IHS Markit branding available.
  3. Start offering the CPL3 via CDN
  4. Start again with regular updates and communication
    1. We’re shooting to update the npm package on a regular basis of every other Friday – starting with our first target of October 13th, 2017 as the first update that makes IHS Markit branding available.
    2. The website will update as often as possible as we make fixes and lay out our roadmap. This will make it the best place to get info on the CPL3 (other than contacting me directly, but I’ll be doing my best to keep information and communication on the website up to date).
    3. We’ve created a slack channel to improve communication. The one we created has run into some technical issues that may necessitate that we create another. We’ll be posting information about it to the website ASAP. The goal is to allow anyone that has an “@ihsmarkit.com” type of email to join the channel. This is currently working for some and not for others – thus the possible need to make a change.
    4. We’ll also be starting up regular “office hours” forum meetings via WebEx soon. This will set up a regularly scheduled forum for CPL3 users (customers) to ask any questions and get personal help as they need it. We’ll also be discussing roadmap changes here to make sure we’re planning to deliver what our customers need. Please contact me if you want to be included or notified about this meeting.
  5. Bookmarkable URLs on CPL3 site
  6. Fix the Patterns section of the CPL3 site so that code snippets are once again present. This will make it easy for developers to copy the patterns they need and paste them into their projects.
  7. Numerous bug fixes and small updates to the website
  8. Start the process of removing direct dependency on the Twitter Bootstrap and jQuery. See the explanation below for more details on why we're making this change and how it will affect you.

Major customer complaints, issues, and our plan for action New Feedback Roadmap posted October 4th, 2017 by Michael Witt

Many customers have complained about the following:

  1. Adoption issues for older or existing projects
    1. Comments like
      1. "Starting up with the CPL3 leaks styles out and breaks existing styles on my website. We can't take on fixing those issues right now."
      2. "We need a way to bring the CPL3 into our project that controls what it affects."
      3. "Can we start using it “piece-meal” within our project to target only the pieces we want it to affect?"
      4. "There's no way we're bringing in all that just to add a header and footer to our project."
    2. Relevant information
      1. The current implementation of CPL3 is close to a standard version of the Twitter Bootstrap with IHS branding colors applied and a few minor tweaks to the styles. The CPL3 mostly offers CSS types of benefits at the moment and our JavaScript is essentially straight from Bootstrap. The CPL3 also offers a few patterns and components that are unique to the CPL3, but not many.
      2. This is a major concern that the majority of our current customers (and most notably our potential new customers) would like us to address.
        1. Action Thus, we have started work to “scope” the entirety of CPL3 styles to only apply to HTML that is nested within an HTML element with the class name of cpl-scope. In many cases this can be as easy as adding the cpl-scope class to the root level html element of your project (something like <html class=”cpl-scope”>).
        2. Breaking change Existing projects that use the CPL3 will have to add the cpl-scope class somewhere in their HTML to continue to be affected by the CPL3's styles. This will be necessary from the October 13th, 2017 npm package release moving forward. Our hope is that this change can even be made by developers in advance of updating the npm package used in their project as it will likely have no effect until the update is made. Then the update should come off without a hitch.
        3. Breaking change Doing this requires us to break our current ability to work with any version 3.x implementation of the Twitter Bootstrap. This is because most of the styles that are leaking out are actually coming from Bootstrap styles. This will mean that we will start encouraging our customers to link to the version of the Twitter Bootstrap that is included in our npm package. Most of our customers are already doing this and do not have specific needs above the version of Bootstrap that is we include. Please let us know if your project has additional needs or concerns around this limitation. Our hope is to ramp up development progress to the point that our offering out-paces Bootstrap improvements and better meets your needs with the customizations that you specifically need.
        4. Transition info This work is mostly complete and working on our Dev site. The plan is to release it as part of the upcoming October 13th npm package release. Thus, CPL3 customers will need to start using the version of Bootstrap included in the CPL3 to get this benefit. Linking to another version of Bootstrap will likely continue to work (with the side-effect of leaking styles) for a little while, but the changes that we make from this point forward will likely start causing issues at some point. We expect to provide support for this issue and plan to address these issues with all of our customers (helping teams make the necessary changes) over the next 2 to 6 months. If this issue causes problems for your project please stop updating as new CPL3 versions are released, use the latest version of the CPL3 that you can without issues, and contact Michael Witt for help in resolving your issues.
  2. Direct dependency on the Twitter Bootstrap and its inherent dependency on jQuery
    1. Comments like
      1. "The Twitter Bootstrap is so heavy. And we're only going to be adding a header and footer because it's being forced upon us. We won't use anything else in the CPL."
      2. "After looking at it we can't add another dependency of Bootstrap (and jQuery) to our project. It will confilict with …"
      3. "I don't like Bootstrap. Why is it a dependency? Wouldn't it be better to use …?"
      4. "What Bootstrap features are people actually using anyway? All of that is so simple people aren't really using it to make anything meaningful are they?"
    2. Relevant information
      1. By directly using the Twitter Bootstrap the CPL3 also:
        1. Starts with a “mobile first” approach
        2. Can make use of the Bootstrap layout grids
          1. Quick for developers to learn and use
          2. But also get the Bootstrap’s 12 column layout grid limitations and problems with nested grids, floated divs, and flex-box quirks.
            1. Design limitations
            2. Only the simplest layouts work well out of the box. Most require serious tweaking to get right.
        3. Benefits from the popularity of Bootstrap and many developer’s familiarity with it. Specifically:
          1. Quick developer adoption
          2. Bootstrap’s naming conventions that provide functionality on a base (.alert) and modifier (.alert-success) class basis.
          3. Layout grid naming conventions - .container .row .col-xs-12
          4. A core set of Bootstrap Components that can be used as building blocks in the construction of more complex components, patterns, and applications.
          5. Lots of documentation, tools, and information on the internet about working with Bootstrap.
          6. Less files are available – developers have direct access to Less mixins and other advanced tooling built into Bootstrap if needed
          7. Adds a dependency on jQuery
          8. Adds Bootstrap’s file size and weight
          9. Adds Bootstrap’s flexibility
          10. Must override unwanted styles that are built into Bootstrap
    3. Action We’re planning a structured change away from direct dependency on Bootstrap
      1. We’re going further than Bootstrap ever intended to go. Enhancing, improving, trimming down, refining, and otherwise breaking away from the limitations of Bootstrap.

        We will make every effort to maintain as much compatibility with the Twitter Bootstrap as possible. We will hold open discussions before making these changes so that their negative impact can be reduced, eliminated, or otherwise negotiated with our customers. We expect to find a way to work with you that still allows us to move forward.

        Customer feedback is a major driver for where we expect to go with this. Please voice any concerns you have so that we may act accordingly.
        1. We’re keeping:
          1. Ability to (optionally) use Bootstrap layout grids (if they’re a good fit for your project). We expect to add the option to choose other layout grid systems as alternatives.
          2. Most of the benefits of developer’s knowledge of Bootstrap:
            1. Quick developer adoption
            2. Backwards compatibility with Bootstrap’s classes and ids. Exceptions will be noted on website
            3. Bootstrap’s naming conventions for base and modifier classes.
            4. Core Bootstrap (based) components – possibly enhanced, improved, trimmed down, tested, etc.
            5. Most documentation, tools, and internet information will continue to apply – Exceptions will be noted on website
            6. Less files still available
            7. Bootstrap’s flexibility
        2. We’re removing, replacing, or reducing:
          1. Removing direct dependency on Twitter Bootstrap
          2. Removing direct dependency on jQuery – all jQuery specific JavaScript will be rewritten in vanilla JavaScript
          3. Paring down core styles so that they need to be overridden less and add less weight to the CPL3
          4. Reducing the need to bring in the whole Bootstrap library – moving toward an “a-la-carte” type of use case.
          5. Replacing Bootstrap’s styles related to spacing. They are not consistent or well designed. Our design will create a standard that we will use throughout the CPL3. This standard should be easy for your developers and designers to remember and apply.
        3. Expected additional benefits:
          1. Speed and file size improvements
          2. Better look and feel - stronger branding, more unified, more flexible
          3. Easier for us to improve the CPL3
          4. Improved framework for applying and understanding spacing in CPL3 designs – padding, margin, border sizes, line-heights, white-space use, etc.
